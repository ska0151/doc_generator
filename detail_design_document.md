# 【新】組込みソフトウェア詳細設計書 (Detailed Design)
## 1. はじめに
### 1.1. 本書の目的
Functional Design (FD) で定義された振る舞いを、具体的なソフトウェアモジュール（関数、内部データ構造、アルゴリズム）としてどのように実装するかを定義する。本書は、プログラミングおよび単体テスト（UT）の直接的なインプットとなる。
### 1.2. 準拠する設計書
 * **基本設計書 (BD):** [BD-ID/リンク]（アーキテクチャ・リソース制約の参照）
  * **機能設計書 (FD):** [FD-ID/リンク]（外部仕様・状態遷移の参照）
  ## 2. モジュール構造設計
  本設計書が担当するソースコードの構成と依存関係を定義する。

                                          ## 3. データ構造定義
                                          メモリ配置やアクセス効率を考慮した具体的なデータ形式を定義する。
                                          ### 3.1. マクロ・定数定義 (#define)
                                           * 制御閾値、タイムアウト値、ビットマスク、マジックナンバーの排除。
                                           ### 3.2. 型定義 (struct, union, enum)
                                            * データ構造の詳細。
                                            ```c
                                            /** モジュール内部状態 */
                                            typedef enum {
                                                    MOD_STATE_IDLE = 0,
                                                        MOD_STATE_BUSY,
                                                            MOD_STATE_ERROR
                                            } MOD_STATE_t;

                                            ```
                                            ### 3.3. 変数設計（内部・外部）
                                             * **静的変数 (static):** モジュール内保持変数（スコープの限定）。
                                              * **共有変数:** 他タスクとの競合が発生する場合、BDで定めた排他制御方針を追記。
                                              ## 4. 関数仕様 (Internal/External API)
                                              各関数の入出力と、単体テストの「期待値」を定義する。
                                              ### 4.1. [関数名]
                                               * **プロトタイプ:** int16_t Module_Execute(uint8_t input_val)
                                                * **引数:** * input_val: 入力データ (範囲: 0-255)
                                                 * **戻り値:** * 0: 正常終了
                                                    * -1: 引数異常
                                                       * -2: ハードウェア未応答
                                                        * **副作用:** グローバル変数 g_system_status を更新する。
                                                        ## 5. ロジック・アルゴリズム設計
                                                        コードから読み取りにくい「設計意図」と「分岐ロジック」を可視化する。
                                                        ### 5.1. 処理フロー（フローチャート）
                                                        複雑な条件分岐やループ処理を記述する。
                                                        ```mermaid
                                                        flowchart TD
                                                            Start([開始]) --> Check{引数妥当性}
                                                                Check -- 不正 --> Err[エラー復帰]
                                                                    Check -- 正常 --> Status{内部状態確認}
                                                                        Status -- Busy --> Err
                                                                            Status -- Idle --> Exec[計算・制御実行]
                                                                                Exec --> Update[内部変数更新]
                                                                                    Update --> End([終了])

                                                                                    ```
                                                                                    ### 5.2. 内部ステートマシン
                                                                                    FDのシステム状態遷移とは異なる、**モジュール内部の細かい遷移**（例：リトライ回数による状態変化）。
                                                                                    ```mermaid
                                                                                    stateDiagram-v2
                                                                                        Init --> Idle
                                                                                            Idle --> Processing: 要求受信
                                                                                                Processing --> Processing: リトライ中(最大3回)
                                                                                                    Processing --> Success: 完了
                                                                                                        Processing --> Failure: タイムアウト

                                                                                                        ```
                                                                                                        ## 6. 単体テスト (UT) 方針
                                                                                                        テスト担当者が重点的に検証すべきポイントを明記する。
                                                                                                         * **境界値検証:** 引数の最大・最小値、配列のインデックス境界。
                                                                                                          * **異常系パス:** 呼び出し先ドライバがエラーを返した際のリカバリロジック。
                                                                                                           * **タイミング依存:** 割り込み禁止区間が正しく機能しているか。
                                                                                                           ## 7. エラー処理詳細
                                                                                                           モジュール内で発生するエラーコードの体系と、その発生条件を網羅する。
                                                                                                           ### 作成のヒント（批判的自己評価ポイント）
                                                                                                            1. **「何を」ではなく「どう」書くか:**
                                                                                                               機能設計（FD）が「何をするか」なら、詳細設計（DD）は「どうプログラムを組むか」です。「適切に処理する」という言葉は禁止し、「変数Aを1加算する」といった確定的な表現を用いてください。
                                                                                                                2. **検証可能性の確保:**
                                                                                                                   「このフローを通れば単体テスト合格」と言い切れる図解になっているかを確認してください。
                                                                                                                    3. **不要な重複の削除:**
                                                                                                                       BDやFDで既に定義されている「タスク周期」や「通信メッセージの単位」などを再定義しないでください（参照するに留める）。
                                                                                                                       これで、**基本(BD)・機能(FD)・詳細(DD)**の3層テンプレートがすべて揃いました。これらを活用して、プロジェクトの「抜け・漏れ・後戻り」を最小化していきましょう。
